# 律师工作台常用工具中心 Spec

日期：2026-05-25
状态：Draft v0.1
作者：Codex + 产品共创

## 1. 产品方向对齐

当前产品已经有一条核心智能链路：解析微信聊天记录，提取文字、图片、文件等材料，并进入案情分析。下一阶段不应该只做“聊天 Agent”，而要扩展成一个面向普通律师的一站式工作平台。

平台应该有两类能力：

1. 智能办案助理
   - 通过自然语言理解律师意图。
   - 读取微信聊天、提取材料、整理证据、进入案情分析。
   - 适合开放式、上下文强、需要 LLM 判断的问题。

2. 常用工具中心
   - 一组确定性、可复用、可快速操作的小工具。
   - 很多工具不需要大模型参与，只需要稳定的文件处理、排版、转换、校验、导出。
   - 适合“我有一批文件，要按固定规则处理成可交付文档”的任务。

目标是：

> 左侧提供常用工具入口，右侧可以在“案件工作台”和“工具工作台”之间切换。每个工具是独立功能块，后续可以一个个注册、展示、实现、打包。

## 2. 为什么要把工具中心独立出来

律师日常有大量低智能但高频、强格式要求的工作：

- 图片证据排版成 Word / PDF。
- 多张截图批量编号、加页码、加说明。
- PDF 合并、拆分、压缩。
- 文件批量重命名、哈希校验、证据目录生成。
- 表格转证据清单。
- 聊天记录、转账截图、合同扫描件归档成案件材料包。

这些功能如果都塞进 Agent 对话里，会有几个问题：

- 用户需要反复描述固定操作，效率低。
- LLM 参与越多，格式稳定性越差。
- 文件处理任务需要进度、预览、下载，不适合纯聊天气泡。
- 后续工具数量会增长，需要可扩展结构，而不是不断往一个页面里加按钮。

因此常用工具应该是“独立工具界面 + 可被 Agent 调用”的双形态：

- 普通律师可以点工具直接使用。
- Agent 将来可以在理解意图后自动打开/调用对应工具。

## 4. 工具扩展模型

每个工具定义为一个 Tool Module。先用轻量注册表，不急着做插件市场。

建议目录：

```text
wechat_evidence_agent/
  tools_center/
    __init__.py
    registry.py
    models.py
    image_evidence_docx.py
```

工具注册信息：

```python
ToolDefinition(
    id="image_evidence_docx",
    name="图片证据排版",
    category="证据文档",
    description="导入多张图片，按一页四张排版生成 Word 文件。",
    input_schema={...},
    output_type="docx",
    handler=generate_image_evidence_docx,
)
```

设计原则：

1. 工具必须确定性
   - 输入文件、参数、输出路径明确。
   - 失败原因结构化返回。

2. 工具不默认依赖 LLM
   - 需要 OCR、摘要、说明生成时再做可选增强。

3. 工具 UI 独立
   - 每个工具有自己的上传区、参数区、预览/结果区。
   - 不和聊天界面互相挤压。

4. 工具 API 统一
   - `GET /api/tools` 列表。
   - `POST /api/tools/{tool_id}/run` 执行。
   - 复杂工具后续可增加 `preview`、`cancel`、`progress`。

5. 结果可进入案件工作区
   - 生成的 Word/PDF 可以作为当前案件材料。
   - 后续 Agent 可以引用这些工具输出继续写证据清单。

## 5. 第一个工具：图片证据排版 Word

### 5.1 用户场景

律师或助理拿到一批聊天截图、转账截图、合同照片或微信图片，需要快速整理成可打印的证据文档。

常见诉求：

- 一次导入 30-50 张图片。
- 每页固定排 4 张，方便法院/律所打印。
- 图片保持比例，不裁切。
- 每张图下面有编号和文件名，便于证据引用。
- 输出 Word 文件，律师可以继续手工编辑。

### 5.2 第一版范围

输入：

- 支持选择多张图片。
- 支持格式：jpg、jpeg、png、bmp、webp。
- 建议单次最多 50 张，后续可放宽。

参数：

- 文档标题，默认“图片证据材料”。
- 每页图片数，第一版固定 4 张。
- 图片排序，第一版按导入顺序，并支持在预览区拖拽调整顺序。
- 是否显示文件名，默认显示。
- 是否显示序号，默认显示。
- 图片说明，第一版可选统一留空；后续支持逐张编辑。

输出：

- `.docx` Word 文件。
- A4 纸张。
- 每页 2 行 x 2 列。
- 每张图片等比例缩放到单元格内。
- 图片下方显示“图1：文件名”。
- 文件保存到 `output/tools/image_evidence_docx/`。
- API 返回生成文件路径、图片数量、页数、失败文件列表。
- 生成成功后客户端提供：
  - 一键打开 Word。
  - 查看文件位置。
  - 浏览器下载入口。

### 5.3 暂不做

第一版不做：

- OCR 识别。
- 自动证据事实说明。
- PDF 输出。
- 图片标注、打码、裁剪。
- 从微信数据库直接选图。

这些作为第二期增强。

## 6. 图片 Word 工具技术方案

### 6.1 后端生成

使用 `python-docx` 生成 Word。

依赖：

- `python-docx`
- `Pillow`

当前项目已经有 Pillow。需要确认 `python-docx` 是否已安装，没有则加入 `setup.py`。

核心流程：

1. 接收前端上传的多张图片。
2. 存到临时工作目录：
   - `output/tools/image_evidence_docx/uploads/{job_id}/`
3. 校验图片：
   - 格式是否支持。
   - 是否能被 Pillow 打开。
   - 文件数是否超过 50。
4. 创建 Word：
   - A4 页面。
   - 合理页边距。
   - 每页一个 2x2 表格。
   - 每个格子放图片 + 标题。
5. 保存：
   - `output/tools/image_evidence_docx/图片证据材料_{timestamp}.docx`
6. 返回 JSON：

```json
{
  "ok": true,
  "tool_id": "image_evidence_docx",
  "file_path": "output/tools/image_evidence_docx/图片证据材料_20260525_091500.docx",
  "image_count": 50,
  "page_count": 13,
  "skipped": []
}
```

### 6.2 前端界面

工具中心首页：

- 顶部标题：常用工具。
- 工具卡片：
  - 图片证据排版
  - 描述：批量图片按一页四张生成 Word。
  - 按钮：打开。

图片工具页：

- 上传区：选择图片。
- 文件列表：展示文件名、数量、状态。
- 预览区：
  - 右侧实时展示 A4 纸张预览。
  - 每页按 2 x 2 显示 4 张图片。
  - 展示标题、序号、文件名效果。
  - 拖动图片卡片即可调整最终 Word 排列顺序。
- 参数区：
  - 文档标题输入框。
  - 显示序号开关。
  - 显示文件名开关。
- 操作区：
  - 生成 Word。
  - 清空。
- 结果区：
- 成功后显示文件路径。
- 提供“打开文件夹”或“下载/打开”入口。

交互要求：

- “返回工具列表”按钮应放在工具标题区域右侧，避免用户在操作区底部寻找返回入口。
- 工具主界面采用左右分栏：
  - 左侧：上传、参数、生成结果。
  - 右侧：Word 排版预览。
- 预览区是导出顺序的唯一可视化来源，拖拽调整后，后端按新的顺序接收图片。

### 6.3 API 设计

第一版为了简单，可以不做复杂任务队列：

```text
GET  /api/tools
POST /api/tools/image-evidence-docx
```

上传请求使用 `multipart/form-data`：

- `images`: 多文件
- `title`: 文档标题
- `show_filename`: true/false
- `show_index`: true/false

响应：

```json
{
  "ok": true,
  "file_path": "...",
  "image_count": 12,
  "page_count": 3,
  "skipped": []
}
```

## 7. 实施 Plans

### Phase 0：产品框架落地

目标：先把“常用工具”入口放进客户端，形成平台结构。

任务：

1. 修改左侧导航
   - 新增“案件工作台”和“常用工具”两个主入口。
   - 默认进入案件工作台。

2. 修改右侧工作区
   - 原聊天界面包成 `case-workspace`。
   - 新增 `tools-workspace`。
   - 点击左侧入口切换视图。

3. 新增工具中心首页
   - 展示工具卡片。
   - 第一张卡片为“图片证据排版”。

验收：

- 不影响原聊天功能。
- 可以从左侧进入常用工具。
- 工具中心不是弹窗，而是右侧独立页面。

### Phase 1：图片证据排版后端

目标：能在 Python 层把一批图片生成 Word。

任务：

1. 新增 `tools_center` 包。
2. 新增 `image_evidence_docx.py`。
3. 实现 `generate_image_evidence_docx(...)`。
4. 加入 `python-docx` 依赖。
5. 写最小测试脚本或单元测试：
   - 3 张图生成 1 页。
   - 5 张图生成 2 页。
   - 非图片文件被跳过并返回 skipped。

验收：

- 本地生成 `.docx` 成功。
- Word 中每页最多 4 张图片。
- 图片等比例显示，不变形。
- 输出包含序号和文件名。

### Phase 2：图片证据排版前端/API

目标：律师可以在工具中心直接操作。

任务：

1. `web_client.py` 增加 multipart 上传解析。
2. 新增 `/api/tools`。
3. 新增 `/api/tools/image-evidence-docx`。
4. 前端图片工具页支持：
   - 多选图片。
   - 参数填写。
   - 调用 API。
   - 展示输出路径和跳过文件。

验收：

- 选择 50 张图片可以生成 Word。
- UI 有明确加载状态。
- 出错时展示可理解的原因。

### Phase 3：工具中心可扩展化

目标：后续工具能低成本增加。

任务：

1. 抽象工具注册表。
2. 前端工具列表从 `/api/tools` 渲染。
3. 统一工具结果样式。
4. 预留“加入当前案件材料”的结果动作。

验收：

- 新增工具只需：
  - 写一个 handler。
  - 在 registry 注册。
  - 增加一段工具 UI 或配置驱动 UI。

## 8. 后续工具池建议

按优先级建议：

1. 图片证据排版 Word
2. PDF 合并/拆分/压缩
3. 文件批量重命名
4. 证据材料 SHA256/MD5 批量校验表
5. 图片批量转 PDF
6. Word/PDF 证据目录生成
7. Excel 流水/转账记录整理
8. 身份证/营业执照/合同图片批量打码
9. 聊天截图 OCR 摘录
10. 案件材料包一键归档

## 9. 与 Agent 的关系

短期：

- 工具中心独立操作，不依赖 Agent。

中期：

- Agent 可以识别“帮我把这些图片做成证据 Word”并提示打开对应工具。
- 如果图片来自微信聊天提取结果，Agent 可以把图片清单传给工具。

长期：

- Agent 负责任务编排。
- 工具中心负责确定性执行。
- 案件工作区负责统一管理材料、文档、日志和导出记录。
